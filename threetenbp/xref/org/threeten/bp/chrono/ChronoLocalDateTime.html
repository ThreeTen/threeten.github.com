51  /**
52   * A date-time without a time-zone in an arbitrary chronology, intended
53   * for advanced globalization use cases.
54   * <p>
55   * <b>Most applications should declare method signatures, fields and variables
56   * as {@link LocalDateTime}, not this interface.</b>
57   * <p>
58   * A {@code ChronoLocalDateTime} is the abstract representation of a local date-time
59   * where the {@code Chrono chronology}, or calendar system, is pluggable.
60   * The date-time is defined in terms of fields expressed by {@link TemporalField},
61   * where most common implementations are defined in {@link ChronoField}.
62   * The chronology defines how the calendar system operates and the meaning of
63   * the standard fields.
64   *
65   * <h4>When to use this interface</h4>
66   * The design of the API encourages the use of {@code LocalDateTime} rather than this
67   * interface, even in the case where the application needs to deal with multiple
68   * calendar systems. The rationale for this is explored in detail in {@link ChronoLocalDate}.
69   * <p>
70   * Ensure that the discussion in {@code ChronoLocalDate} has been read and understood
71   * before using this interface.
72   *
73   * <h3>Specification for implementors</h3>
74   * This interface must be implemented with care to ensure other classes operate correctly.
75   * All implementations that can be instantiated must be final, immutable and thread-safe.
76   * Subclasses should be Serializable wherever possible.
77   *
78   * @param <D> the date type
79   */
80  public interface ChronoLocalDateTime<D extends ChronoLocalDate<D>>
81          extends  Temporal, TemporalAdjuster, Comparable<ChronoLocalDateTime<?>> {

83     /**
84       * Comparator for two {@code ChronoLocalDateTime} instances ignoring the chronology.
85       * <p>
86       * This method differs from the comparison in {@link #compareTo} in that it
87       * only compares the underlying date and not the chronology.
88       * This allows dates in different calendar systems to be compared based
89       * on the time-line position.
90       *
91       * @see #isAfter
92       * @see #isBefore
93       * @see #isEqual
94       */
95      Comparator<ChronoLocalDateTime<?>> DATE_TIME_COMPARATOR =
96              new Comparator<ChronoLocalDateTime<?>>() {
97          @Override
98          public int compare(ChronoLocalDateTime<?> datetime1, ChronoLocalDateTime<?> datetime2) {
99              int cmp = Long.compare(datetime1.toLocalDate().toEpochDay(), datetime2.toLocalDate().toEpochDay());
100             if (cmp == 0) {
101                 cmp = Long.compare(datetime1.toLocalTime().toNanoOfDay(), datetime2.toLocalTime().toNanoOfDay());
102             }
103             return cmp;
104         }
105     };

148     /**
149      * Combines this time with a time-zone to create a {@code ChronoZonedDateTime}.
150      * <p>
151      * This returns a {@code ChronoZonedDateTime} formed from this date-time at the
152      * specified time-zone. The result will match this date-time as closely as possible.
153      * Time-zone rules, such as daylight savings, mean that not every local date-time
154      * is valid for the specified zone, thus the local date-time may be adjusted.
155      * <p>
156      * The local date-time is resolved to a single instant on the time-line.
157      * This is achieved by finding a valid offset from UTC/Greenwich for the local
158      * date-time as defined by the {@link ZoneRules rules} of the zone ID.
159      *<p>
160      * In most cases, there is only one valid offset for a local date-time.
161      * In the case of an overlap, where clocks are set back, there are two valid offsets.
162      * This method uses the earlier offset typically corresponding to "summer".
163      * <p>
164      * In the case of a gap, where clocks jump forward, there is no valid offset.
165      * Instead, the local date-time is adjusted to be later by the length of the gap.
166      * For a typical one hour daylight savings change, the local date-time will be
167      * moved one hour later into the offset typically corresponding to "summer".
168      * <p>
169      * To obtain the later offset during an overlap, call
170      * {@link ChronoZonedDateTime#withLaterOffsetAtOverlap()} on the result of this method.
171      *
172      * @param zone  the time-zone to use, not null
173      * @return the zoned date-time formed from this date-time, not null
174      */
175     ChronoZonedDateTime<D> atZone(ZoneId zone);

203     /**
204      * Compares this date-time to another date-time, including the chronology.
205      * <p>
206      * The comparison is based first on the underlying time-line date-time, then
207      * on the chronology.
208      * It is "consistent with equals", as defined by {@link Comparable}.
209      * <p>
210      * For example, the following is the comparator order:
211      * <ol>
212      * <li>{@code 2012-12-03T12:00 (ISO)}</li>
213      * <li>{@code 2012-12-04T12:00 (ISO)}</li>
214      * <li>{@code 2555-12-04T12:00 (ThaiBuddhist)}</li>
215      * <li>{@code 2012-12-05T12:00 (ISO)}</li>
216      * </ol>
217      * Values #2 and #3 represent the same date-time on the time-line.
218      * When two values represent the same date-time, the chronology ID is compared to distinguish them.
219      * This step is needed to make the ordering "consistent with equals".
220      * <p>
221      * If all the date-time objects being compared are in the same chronology, then the
222      * additional chronology stage is not required and only the local date-time is used.
223      *
224      * @param other  the other date-time to compare to, not null
225      * @return the comparator value, negative if less, positive if greater
226      */
