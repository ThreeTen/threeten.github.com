64  /**
65   * A calendar system, used to organize and identify dates.
66   * <p>
67   * The main date and time API is built on the ISO calendar system.
68   * This class operates behind the scenes to represent the general concept of a calendar system.
69   * For example, the Japanese, Minguo, Thai Buddhist and others.
70   * <p>
71   * Most other calendar systems also operate on the shared concepts of year, month and day,
72   * linked to the cycles of the Earth around the Sun, and the Moon around the Earth.
73   * These shared concepts are defined by {@link ChronoField} and are availalbe
74   * for use by any {@code Chrono} implementation:
75   * <pre>
76   *   LocalDate isoDate = ...
77   *   ChronoLocalDate&lt;ThaiBuddhistChrono&gt; minguoDate = ...
78   *   int isoYear = isoDate.get(ChronoField.YEAR);
79   *   int thaiYear = thaiDate.get(ChronoField.YEAR);
80   * </pre>
81   * As shown, although the date objects are in different calendar systems, represented by different
82   * {@code Chrono} instances, both can be queried using the same constant on {@code ChronoField}.
83   * For a full discussion of the implications of this, see {@link ChronoLocalDate}.
84   * In general, the advice is to use the known ISO-based {@code LocalDate}, rather than
85   * {@code ChronoLocalDate}.
86   * <p>
87   * While a {@code Chrono} object typically uses {@code ChronoField} and is based on
88   * an era, year-of-era, month-of-year, day-of-month model of a date, this is not required.
89   * A {@code Chrono} instance may represent a totally different kind of calendar system,
90   * such as the Mayan.
91   * <p>
92   * In practical terms, the {@code Chrono} instance also acts as a factory.
93   * The {@link #of(String)} method allows an instance to be looked up by identifier,
94   * while the {@link #ofLocale(Locale)} method allows lookup by locale.
95   * <p>
96   * The {@code Chrono} instance provides a set of methods to create {@code ChronoLocalDate} instances.
97   * The date classes are used to manipulate specific dates.
98   * <p><ul>
99   * <li> {@link #dateNow() dateNow()}
100  * <li> {@link #dateNow(Clock) dateNow(clock)}
101  * <li> {@link #dateNow(ZoneId) dateNow(zone)}
102  * <li> {@link #date(int, int, int) date(yearProleptic, month, day)}
103  * <li> {@link #date(Era, int, int, int) date(era, yearOfEra, month, day)}
104  * <li> {@link #dateYearDay(int, int) dateYearDay(yearProleptic, dayOfYear)}
105  * <li> {@link #dateYearDay(Era, int, int) dateYearDay(era, yearOfEra, dayOfYear)}
106  * <li> {@link #date(TemporalAccessor) date(TemporalAccessor)}
107  * </ul><p>
108  *
109  * <p id="addcalendars">Adding New Calendars</p>
110  * The set of available chronologies can be extended by applications.
111  * Adding a new calendar system requires the writing of an implementation of
112  * {@code Chrono}, {@code ChronoLocalDate} and {@code Era}.
113  * The majority of the logic specific to the calendar system will be in
114  * {@code ChronoLocalDate}. The {@code Chrono} subclass acts as a factory.
115  * <p>
116  * To permit the discovery of additional chronologies, the {@link java.util.ServiceLoader ServiceLoader}
117  * is used. A file must be added to the {@code META-INF/services} directory with the
118  * name 'org.threeten.bp.chrono.Chrono' listing the implementation classes.
119  * See the ServiceLoader for more details on service loading.
120  * For lookup by id or calendarType, the system provided calendars are found
121  * first followed by application provided calendars.
122  * <p>
123  * Each chronology must define a chronology ID that is unique within the system.
124  * If the chronology represents a calendar system defined by the
125  * <em>Unicode Locale Data Markup Language (LDML)</em> specification then that
126  * calendar type should also be specified.
127  *
128  * <h3>Specification for implementors</h3>
129  * This class must be implemented with care to ensure other classes operate correctly.
130  * All implementations that can be instantiated must be final, immutable and thread-safe.
131  * Subclasses should be Serializable wherever possible.
132  */
133 public abstract class Chronology implements Comparable<Chronology> {

135     /**
136      * Map of available calendars by ID.
137      */
138     private static final ConcurrentHashMap<String, Chronology> CHRONOS_BY_ID;
139     /**
140      * Map of available calendars by calendar type.
141      */
142     private static final ConcurrentHashMap<String, Chronology> CHRONOS_BY_TYPE;
143     static {
144         // TODO: defer initialization?
145         ConcurrentHashMap<String, Chronology> ids = new ConcurrentHashMap<>();
146         ConcurrentHashMap<String, Chronology> types = new ConcurrentHashMap<>();
147         ServiceLoader<Chronology> loader =  ServiceLoader.load(Chronology.class);
148         for (Chronology chrono : loader) {
149             ids.putIfAbsent(chrono.getId(), chrono);
150             String type = chrono.getCalendarType();
151             if (type != null) {
152                 types.putIfAbsent(type, chrono);
153             }
154         }
155         CHRONOS_BY_ID = ids;
156         CHRONOS_BY_TYPE = types;
157     }

160     /**
161      * Obtains an instance of {@code Chrono} from a temporal object.
162      * <p>
163      * A {@code TemporalAccessor} represents some form of date and time information.
164      * This factory converts the arbitrary temporal object to an instance of {@code Chrono}.
165      * If the specified temporal object does not have a chronology, {@link IsoChronology} is returned.
166      * <p>
167      * The conversion will obtain the chronology using {@link TemporalQueries#chronology()}.
168      * <p>
169      * This method matches the signature of the functional interface {@link TemporalQuery}
170      * allowing it to be used in queries via method reference, {@code Chrono::from}.
171      *
172      * @param temporal  the temporal to convert, not null
173      * @return the chronology, not null
174      * @throws DateTimeException if unable to convert to an {@code Chrono}
175      */
176     public static Chronology from(TemporalAccessor temporal) {
177         Objects.requireNonNull(temporal, "temporal");
178         Chronology obj = temporal.query(TemporalQueries.chronology());
179         return (obj != null ? obj : IsoChronology.INSTANCE);
180     }

182     //-----------------------------------------------------------------------
183     /**
184      * Obtains an instance of {@code Chronology} from a locale.
185      * <p>
186      * This returns a {@code Chronology} based on the specified locale,
187      * typically returning {@code IsoChronology}. Other calendar systems
188      * are only returned if they are explicitly selected within the locale.
189      * <p>
190      * The {@link Locale} class provide access to a range of information useful
191      * for localizing an application. This includes the language and region,
192      * such as "en-GB" for English as used in Great Britain.
193      * <p>
194      * The {@code Locale} class also supports an extension mechanism that
195      * can be used to identify a calendar system. The mechanism is a form
196      * of key-value pairs, where the calendar system has the key "ca".
197      * For example, the locale "en-JP-u-ca-japanese" represents the English
198      * language as used in Japan with the Japanese calendar system.
199      * <p>
200      * This method finds the desired calendar system by in a manner equivalent
201      * to passing "ca" to {@link Locale#getUnicodeLocaleType(String)}.
202      * If the "ca" key is not present, then {@code IsoChronology} is returned.
203      * <p>
204      * Note that the behavior of this method differs from the older
205      * {@link java.util.Calendar#getInstance(Locale)} method.
206      * If that method receives a locale of "th_TH" it will return {@code BuddhistCalendar}.
207      * By contrast, this method will return {@code IsoChronology}.
208      * Passing the locale "th-TH-u-ca-buddhist" into either method will
209      * result in the Thai Buddhist calendar system and is therefore the
210      * recommended approach going forward for Thai calendar system localization.
211      * <p>
212      * A similar, but simpler, situation occurs for the Japanese calendar system.
213      * The locale "jp_JP_JP" has previously been used to access the calendar.
214      * However, unlike the Thai locale, "ja_JP_JP" is automatically converted by
215      * {@code Locale} to the modern and recommended form of "ja-JP-u-ca-japanese".
216      * Thus, there is no difference in behavior between this method and
217      * {@code Calendar#getInstance(Locale)}.
218      *
219      * @param locale  the locale to use to obtain the calendar system, not null
220      * @return the calendar system associated with the locale, not null
221      * @throws DateTimeException if the locale-specified calendar cannot be found
222      */
223     public static Chronology ofLocale(Locale locale) {
224         Objects.requireNonNull(locale, "locale");
225         String type = locale.getUnicodeLocaleType("ca");
226         if (type == null || "iso".equals(type) || "iso8601".equals(type)) {
227             return IsoChronology.INSTANCE;
228         } else {
229             Chronology chrono = CHRONOS_BY_TYPE.get(type);
230             if (chrono == null) {
231                 throw new DateTimeException("Unknown calendar system: " + type);
232             }
233             return chrono;
234         }
235     }

238     /**
239      * Obtains an instance of {@code Chrono} from a chronology ID or
240      * calendar system type.
241      * <p>
242      * This returns a chronology based on either the ID or the type.
243      * The {@link #getId() chronology ID} uniquely identifies the chronology.
244      * The {@link #getCalendarType() calendar system type} is defined by the LDML specification.
245      * <p>
246      * The chronology may be a system chronology or a chronology
247      * provided by the application via ServiceLoader configuration.
248      * <p>
249      * Since some calendars can be customized, the ID or type typically refers
250      * to the default customization. For example, the Gregorian calendar can have multiple
251      * cutover dates from the Julian, but the lookup only provides the default cutover date.
252      *
253      * @param id  the chronology ID or calendar system type, not null
254      * @return the chronology with the identifier requested, not null
255      * @throws DateTimeException if the chronology cannot be found
256      */
257     public static Chronology of(String id) {
258         Chronology chrono = CHRONOS_BY_ID.get(id);
259         if (chrono != null) {
260             return chrono;
261         }
262         chrono = CHRONOS_BY_TYPE.get(id);
263         if (chrono != null) {
264             return chrono;
265         }
266         throw new DateTimeException("Unknown chronology: " + id);
267     }

535     /**
536      * Obtains a zoned date-time in this chronology from another temporal object.
537      * <p>
538      * This creates a date-time in this chronology based on the specified {@code TemporalAccessor}.
539      * <p>
540      * This should obtain a {@code ZoneId} using {@link ZoneId#from(TemporalAccessor)}.
541      * The date-time should be obtained by obtaining an {@code Instant}.
542      * If that fails, the local date-time should be used.
543      *
544      * @param temporal  the temporal object to convert, not null
545      * @return the zoned date-time in this chronology, not null
546      * @throws DateTimeException if unable to create the date-time
547      */
548     @SuppressWarnings({ "rawtypes", "unchecked" })
549     public ChronoZonedDateTime<?> zonedDateTime(TemporalAccessor temporal) {
550         try {
551             ZoneId zone = ZoneId.from(temporal);
552             try {
553                 Instant instant = Instant.from(temporal);
554                 return zonedDateTime(instant, zone);
555 
556             } catch (DateTimeException ex1) {
557                 ChronoLocalDateTime cldt = localDateTime(temporal);
