41  /**
42   * Framework-level interface defining read-write access to a temporal object,
43   * such as a date, time, offset or some combination of these.
44   * <p>
45   * This is the base interface type for date, time and offset objects that
46   * are complete enough to be manipulated using plus and minus.
47   * It is implemented by those classes that can provide and manipulate information
48   * as {@link TemporalField fields} or {@link TemporalQuery queries}.
49   * See {@link TemporalAccessor} for the read-only version of this interface.
50   * <p>
51   * Most date and time information can be represented as a number.
52   * These are modeled using {@code TemporalField} with the number held using
53   * a {@code long} to handle large values. Year, month and day-of-month are
54   * simple examples of fields, but they also include instant and offsets.
55   * See {@link ChronoField} for the standard set of fields.
56   * <p>
57   * Two pieces of date/time information cannot be represented by numbers,
58   * the {@link Chronology chronology} and the {@link ZoneId time-zone}.
59   * These can be accessed via {@link #query(TemporalQuery) queries} using
60   * the static methods defined on {@link TemporalQueries}.
61   * <p>
62   * This interface is a framework-level interface that should not be widely
63   * used in application code. Instead, applications should create and pass
64   * around instances of concrete types, such as {@code LocalDate}.
65   * There are many reasons for this, part of which is that implementations
66   * of this interface may be in calendar systems other than ISO.
67   * See {@link ChronoLocalDate} for a fuller discussion of the issues.
68   *
69   * <h3>When to implement</h3>
70   * <p>
71   * A class should implement this interface if it meets three criteria:
72   * <p><ul>
73   * <li>it provides access to date/time/offset information, as per {@code TemporalAccessor}
74   * <li>the set of fields are contiguous from the largest to the smallest
75   * <li>the set of fields are complete, such that no other field is needed to define the
76   *  valid range of values for the fields that are represented
77   * </ul><p>
78   * <p>
79   * Four examples make this clear:
80   * <p><ul>
81   * <li>{@code LocalDate} implements this interface as it represents a set of fields
82   *  that are contiguous from days to forever and require no external information to determine
83   *  the validity of each date. It is therefore able to implement plus/minus correctly.
84   * <li>{@code LocalTime} implements this interface as it represents a set of fields
85   *  that are contiguous from nanos to within days and require no external information to determine
86   *  validity. It is able to implement plus/minus correctly, by wrapping around the day.
87   * <li>{@code MonthDay}, the combination of month-of-year and day-of-month, does not implement
88   *  this interface.  While the combination is contiguous, from days to months within years,
89   *  the combination does not have sufficient information to define the valid range of values
90   *  for day-of-month.  As such, it is unable to implement plus/minus correctly.
91   * <li>The combination day-of-week and day-of-month ("Friday the 13th") should not implement
92   *  this interface. It does not represent a contiguous set of fields, as days to weeks overlaps
93   *  days to months.
94   * </ul><p>
95   *
96   * <h3>Specification for implementors</h3>
97   * This interface places no restrictions on the mutability of implementations,
98   * however immutability is strongly recommended.
99   * All implementations must be {@link Comparable}.
100  */
101 public interface Temporal extends TemporalAccessor {

103     /**
104      * Checks if the specified unit is supported.
105      * <p>
106      * This checks if the date-time can be queried for the specified unit.
107      * If false, then calling the {@link #plus(TemporalAmount) plus} and {@link #minus(TemporalAmount) minus}
108      * methods will throw an exception.
109      *
110      * <h3>Specification for implementors</h3>
111      * Implementations must check and handle all fields defined in {@link ChronoUnit}.
112      * If the field is supported, then true is returned, otherwise false
113      * <p>
114      * If the field is not a {@code ChronoUnit}, then the result of this method
115      * is obtained by invoking {@code TemporalUnit.isSupportedBy(Temporal)}
116      * passing {@code this} as the argument.
117      * <p>
118      * Implementations must not alter this object.
119      *
120      * @param unit  the unit to check, null returns false
121      * @return true if this date-time can be queried for the unit, false if not
122      */
123     boolean isSupported(TemporalUnit unit);
124 
125     /**
126      * Returns an adjusted object of the same type as this object with the adjustment made.
127      * <p>
128      * This adjusts this date-time according to the rules of the specified adjuster.
129      * A simple adjuster might simply set the one of the fields, such as the year field.
130      * A more complex adjuster might set the date to the last day of the month.
131      * A selection of common adjustments is provided in {@link TemporalAdjusters}.
132      * These include finding the "last day of the month" and "next Wednesday".
133      * The adjuster is responsible for handling special cases, such as the varying
134      * lengths of month and leap years.
135      * <p>
136      * Some example code indicating how and why this method is used:
137      * <pre>
138      *  date = date.with(Month.JULY);        // most key classes implement TemporalAdjuster
139      *  date = date.with(lastDayOfMonth());  // static import from TemporalAdjusters
140      *  date = date.with(next(WEDNESDAY));   // static import from TemporalAdjusters and DayOfWeek
141      * </pre>
142      *
143      * <h3>Specification for implementors</h3>
144      * Implementations must not alter either this object.
145      * Instead, an adjusted copy of the original must be returned.
146      * This provides equivalent, safe behavior for immutable and mutable implementations.
147      *
148      * @param adjuster  the adjuster to use, not null
149      * @return an object of the same type with the specified adjustment made, not null
150      * @throws DateTimeException if unable to make the adjustment
151      * @throws ArithmeticException if numeric overflow occurs
152      */
153     Temporal with(TemporalAdjuster adjuster);
154 
155     /**
156      * Returns an object of the same type as this object with the specified field altered.
157      * <p>
158      * This returns a new object based on this one with the value for the specified field changed.
159      * For example, on a {@code LocalDate}, this could be used to set the year, month or day-of-month.
160      * The returned object will have the same observable type as this object.
161      * <p>
162      * In some cases, changing a field is not fully defined. For example, if the target object is
163      * a date representing the 31st January, then changing the month to February would be unclear.
164      * In cases like this, the field is responsible for resolving the result. Typically it will choose
165      * the previous valid date, which would be the last valid day of February in this example.
166      *
167      * <h3>Specification for implementors</h3>
168      * Implementations must check and handle all fields defined in {@link ChronoField}.
169      * If the field is supported, then the adjustment must be performed.
170      * If unsupported, then a {@code DateTimeException} must be thrown.
171      * <p>
172      * If the field is not a {@code ChronoField}, then the result of this method
173      * is obtained by invoking {@code TemporalField.adjustInto(Temporal, long)}
174      * passing {@code this} as the first argument.
175      * <p>
176      * Implementations must not alter either this object or the specified temporal object.
177      * Instead, an adjusted copy of the original must be returned.
178      * This provides equivalent, safe behavior for immutable and mutable implementations.
179      *
180      * @param field  the field to set in the result, not null
181      * @param newValue  the new value of the field in the result
182      * @return an object of the same type with the specified field set, not null
183      * @throws DateTimeException if the field cannot be set
184      * @throws ArithmeticException if numeric overflow occurs
185      */
186     Temporal with(TemporalField field, long newValue);

189     /**
190      * Returns an object of the same type as this object with an amount added.
191      * <p>
192      * This adjusts this temporal, adding according to the rules of the specified amount.
193      * The amount is typically a {@link Period} but may be any other type implementing
194      * the {@link TemporalAmount} interface, such as {@link Duration}.
195      * <p>
196      * Some example code indicating how and why this method is used:
197      * <pre>
198      *  date = date.plus(period);                  // add a Period instance
199      *  date = date.plus(duration);                // add a Duration instance
200      *  date = date.plus(workingDays(6));          // example user-written workingDays method
201      * </pre>
202      * <p>
203      * Note that calling {@code plus} followed by {@code minus} is not guaranteed to
204      * return the same date-time.
205      *
206      * <h3>Specification for implementors</h3>
207      * Implementations must not alter either this object.
208      * Instead, an adjusted copy of the original must be returned.
209      * This provides equivalent, safe behavior for immutable and mutable implementations.
210      *
211      * @param amount  the amount to add, not null
212      * @return an object of the same type with the specified adjustment made, not null
213      * @throws DateTimeException if the addition cannot be made
214      * @throws ArithmeticException if numeric overflow occurs
215      */
216     Temporal plus(TemporalAmount amount);

218     /**
219      * Returns an object of the same type as this object with the specified period added.
220      * <p>
221      * This method returns a new object based on this one with the specified period added.
222      * For example, on a {@code LocalDate}, this could be used to add a number of years, months or days.
223      * The returned object will have the same observable type as this object.
224      * <p>
225      * In some cases, changing a field is not fully defined. For example, if the target object is
226      * a date representing the 31st January, then adding one month would be unclear.
227      * In cases like this, the field is responsible for resolving the result. Typically it will choose
228      * the previous valid date, which would be the last valid day of February in this example.
229      * <p>
230      * If the implementation represents a date-time that has boundaries, such as {@code LocalTime},
231      * then the permitted units must include the boundary unit, but no multiples of the boundary unit.
232      * For example, {@code LocalTime} must accept {@code DAYS} but not {@code WEEKS} or {@code MONTHS}.
233      *
234      * <h3>Specification for implementors</h3>
235      * Implementations must check and handle all units defined in {@link ChronoUnit}.
236      * If the unit is supported, then the addition must be performed.
237      * If unsupported, then a {@code DateTimeException} must be thrown.
238      * <p>
239      * If the unit is not a {@code ChronoUnit}, then the result of this method
240      * is obtained by invoking {@code TemporalUnit.addTo(Temporal, long)}
241      * passing {@code this} as the first argument.
242      * <p>
243      * Implementations must not alter either this object or the specified temporal object.
244      * Instead, an adjusted copy of the original must be returned.
245      * This provides equivalent, safe behavior for immutable and mutable implementations.
246      *
247      * @param amountToAdd  the amount of the specified unit to add, may be negative
248      * @param unit  the unit of the period to add, not null
249      * @return an object of the same type with the specified period added, not null
250      * @throws DateTimeException if the unit cannot be added
251      * @throws ArithmeticException if numeric overflow occurs
252      */
253     Temporal plus(long amountToAdd, TemporalUnit unit);

277      *
278      * @param amount  the amount to subtract, not null
279      * @return an object of the same type with the specified adjustment made, not null
280      * @throws DateTimeException if the subtraction cannot be made
281      * @throws ArithmeticException if numeric overflow occurs

286      * Returns an object of the same type as this object with the specified period subtracted.

288      * This method returns a new object based on this one with the specified period subtracted.
289      * For example, on a {@code LocalDate}, this could be used to subtract a number of years, months or days.
290      * The returned object will have the same observable type as this object.
291      * <p>
292      * In some cases, changing a field is not fully defined. For example, if the target object is
293      * a date representing the 31st March, then subtracting one month would be unclear.
294      * In cases like this, the field is responsible for resolving the result. Typically it will choose
295      * the previous valid date, which would be the last valid day of February in this example.

298      * then the permitted units must include the boundary unit, but no multiples of the boundary unit.
299      * For example, {@code LocalTime} must accept {@code DAYS} but not {@code WEEKS} or {@code MONTHS}.
300      *
301      * <h3>Specification for implementors</h3>
302      * Implementations must behave in a manor equivalent to the default method behavior.
303      * <p>
304      * Implementations must not alter either this object or the specified temporal object.
305      * Instead, an adjusted copy of the original must be returned.
306      * This provides equivalent, safe behavior for immutable and mutable implementations.
307      *
308      * @param amountToSubtract  the amount of the specified unit to subtract, may be negative
309      * @param unit  the unit of the period to subtract, not null
310      * @return an object of the same type with the specified period subtracted, not null
311      * @throws DateTimeException if the unit cannot be subtracted
312      * @throws ArithmeticException if numeric overflow occurs
313      */
314     Temporal minus(long amountToSubtract, TemporalUnit unit);

316     //-----------------------------------------------------------------------
317     /**
318      * Calculates the period between this temporal and another temporal in
319      * terms of the specified unit.
320      * <p>
321      * This calculates the period between two temporals in terms of a single unit.
322      * The start and end points are {@code this} and the specified temporal.
323      * The result will be negative if the end is before the start.
324      * For example, the period in hours between two temporal objects can be
325      * calculated using {@code startTime.until(endTime, HOURS)}.
326      * <p>
327      * The calculation returns a whole number, representing the number of
328      * complete units between the two temporals.
329      * For example, the period in hours between the times 11:30 and 13:29
330      * will only be one hour as it is one minute short of two hours.
331      * <p>
332      * There are two equivalent ways of using this method.
333      * The first is to invoke this method directly.
334      * The second is to use {@link TemporalUnit#between(Temporal, Temporal)}: 
335      * <pre>
336      *   // these two lines are equivalent
337      *   between = thisUnit.between(start, end);
338      *   between = start.until(end, thisUnit);
339      * </pre>
340      * The choice should be made based on which makes the code more readable. 
341      * <p>
342      * For example, this method allows the number of days between two dates to be calculated:
343      * <pre>
344      *   long daysBetween = DAYS.between(start, end);
345      *   // or alternatively
346      *   long daysBetween = start.until(end, DAYS);
347      * </pre>
348      *
349      * <h3>Specification for implementors</h3>
350      * Implementations must begin by checking to ensure that the input temporal
351      * object is of the same observable type as the implementation.
352      * They must then perform the calculation for all instances of {@link ChronoUnit}.
353      * A {@code DateTimeException} must be thrown for {@code ChronoUnit}
354      * instances that are unsupported.
355      * <p>
356      * If the unit is not a {@code ChronoUnit}, then the result of this method
357      * is obtained by invoking {@code TemporalUnit.between(Temporal, Temporal)}
358      * passing {@code this} as the first argument and the input temporal as
359      * the second argument.
360      * <p>
361      * In summary, implementations must behave in a manner equivalent to this code:
362      * <pre>
363      *  // check input temporal is the same type as this class
364      *  if (unit instanceof ChronoUnit) {
365      *    // if unit is supported, then calculate and return result
366      *    // else throw DateTimeException for unsupported units
367      *  }
368      *  return unit.between(this, endTemporal);
369      * </pre>
370      * <p>
371      * The target object must not be altered by this method.
372      *
373      * @param endTemporal  the end temporal, of the same type as this object, not null
374      * @param unit  the unit to measure the period in, not null
375      * @return the amount of the period between this and the end
376      * @throws DateTimeException if the period cannot be calculated
377      * @throws ArithmeticException if numeric overflow occurs
378      */
379     long until(Temporal endTemporal, TemporalUnit unit);
